== Architecture Constraints

The architecture of the University Management System is influenced by several technical and organizational constraints. These constraints limit the design freedom and must therefore be considered in all major architectural decisions.

=== Technical Constraints

[options="header",cols="1,3"]
|===
|Constraint |Explanation

|Web-based platform
|The system is planned as a web-based platform that can be accessed by students, lecturers, administrators, and finance staff through a browser.

|Integration with existing systems
|The application must integrate with the university’s existing student information system and financial management system.

|Modern web technologies
|The project assumptions already suggest the use of modern web technologies such as React, ASP.NET, and a modern relational database, which limits the technology choices.

|Cloud hosting
|The system is expected to run on secure and scalable cloud infrastructure such as AWS or Azure.
|===

=== Organizational Constraints

[options="header",cols="1,3"]
|===
|Constraint |Explanation

|Semester project scope
|The project is developed in an academic context with limited time and resources, so the architecture must remain realistic and manageable.

|Focus on architecture
|The main focus of the project is on software architecture and documentation, not on full implementation.

|Documentation with arc42
|The architecture has to be documented using the arc42 template, which influences the structure and level of detail of the documentation.

|Collaboration through GitHub
|Project materials are managed in a shared GitHub repository, so documentation and diagrams should use formats that are easy to version and review.
|===

=== Quality and Compliance Constraints

[options="header",cols="1,3"]
|===
|Constraint |Explanation

|Data protection and privacy
|The system processes sensitive information such as grades, attendance records, and personal student data, so privacy and data protection requirements must be considered.

|Different user roles
|The architecture must support different user groups with different access rights, such as students, lecturers, administrators, and finance staff.
|===
